# Правила работы с кодом

Как я пишу код для alexs и как с ним общаюсь.

Общие технические правила кодинга — **`rules/deepseek.md`** (13 пунктов: думать перед кодом, простота, хирургические правки, безопасность, стиль, чекпоинты и т.д.).

Здесь — только то, что относится лично к alexs и не покрыто deepseek.md.

## 1. Как объяснять

- Сначала суть — коротко, парой предложений.
- Потом детали — если нужно, развернуть, но без воды.
- Не оставлять ничего «на усмотрение». Если есть варианты — сказать о них, но предложить лучший.
- Язык — максимально простой и понятный. Без жаргона, эндпоинтов, мидлваров.

## 2. Комментарии в коде

- Минимум комментариев.
- Комментарии — только если без них реально непонятно, что и зачем.
- Простые функции, импорты, очевидные блоки — без комментариев.

## 3. Ссылки для обучения

Давать ссылки на документацию, статьи, примеры. alexs хочет учиться и понимать глубже.

## 4. Тесты

Перед сложными изменениями (более 3 файлов, смена логики, рефакторинг) — делать тест.
- Тест проверяет ключевые сценарии, которые могу случайно сломать
- Тест запускается одной командой — результат виден сразу
- Тест не обязан покрывать всё, только критичное
- Первый тест: `tests/test_vault_storage.py` (11 шагов: создание, обновление, поиск, удаление групп и записей)
